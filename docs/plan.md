## 1. ゴール

154件の都市・観光地について、旅行先提案ロジックが利用できる**客観的事実データ**を「調査 → 設計 → 作成」の3工程で整備する。

最終成果物は次の3点。

| # | 成果物 | 形式・場所 |
| --- | --- | --- |
| 1 | データソース調査レポート | `docs/01_data-source-survey.md` |
| 2 | データ構造設計書（スキーマ定義＋指標マッピング） | `docs/02_schema-design.md` ＋ JSON Schema |
| 3 | 154件分の統一フォーマットデータ | `data/output/` 配下の JSON |

## 2. 決定済みの前提

- **非都市エントリの扱い**：全154件を同一スキーマで整備し、`type` 属性（都市／城・単体スポット／自然地物／航路 など）で区別する。都市向け指標（POI密度など）が該当しないエントリは null を許容する。
- **成果物の置き場所**：本リポジトリ（city-data-builder）に JSON＋Markdown で整備する。`global-bm` への組み込みは後工程。
- **進め方**：工程順（4.1調査 → 4.2設計 → 4.3作成）に全体を進め、各工程完了時にレビュー・確認を挟む。
- **費用**：まずは無料のデータソースのみで構成する。

## 3. 対象154件の整理（フェーズ0：名寄せマスタの作成）

現状の `cities.json` は**日本語カタカナ名の配列のみ**であり、外部データソースと突合するためのキーがない。すべての工程の土台として、まず名寄せマスタを作成する。

### 3.1 やること

1. 154件それぞれに以下を付与した `data/master/places_master.json` を作成する。
   - 安定ID（例：`amsterdam` のようなslug）
   - 日本語名（原本のまま）／英語名／現地語名
   - **Wikidata QID**（以降のデータ結合の主キー。人口・位置・Wikipedia記事・OSM等はすべてQID経由で紐付け可能）
   - 国、緯度経度
   - `type` 分類（暫定案：`city` / `town_village` / `single_poi`（城・遺跡・修道院など） / `natural_feature`（湖・フィヨルド等） / `route`（フェリー航路等））
2. 同定が曖昧なエントリは候補を提示し、確認を仰ぐ。

### 3.2 現時点で確認が必要そうなエントリ（例）

| エントリ | 論点 |
| --- | --- |
| ストックホルム―トゥルク、デンマーク/ノルウェー間フェリー | 「旅行先」ではなく航路。`type: route` として最小限の属性のみ整備する想定でよいか |
| シヨン城、シャンボール、シュノンソー、ビランドリー、ストーンヘンジ 等 | 単体スポット。所在都市（近隣拠点都市）への参照を持たせる想定 |
| ネス湖、ソグネフィヨルド、プリトビチェ、モンサンミッシェル 等 | 自然地物・景勝地。同上 |
| カレー、カーン、ランス、レバント、メリダ、トラー二 等 | 同名地が複数あるため同定結果（Calais / Caen / Reims / Levanto / Mérida(西) / Trani を想定）を確認いただきたい |
| ストックホルム（単体）とストックホルム―トゥルク | 重複的な扱いになるため、後者は航路として分離してよいか |

> マスタ完成時に「日本語名 → 同定結果（英語名・国・QID）」の一覧を提出し、レビューを受けてから次工程へ進む。

## 4. フェーズ1：データソース調査（note.md 4.1）

note.md 記載の候補ソースを対象に、指定の9項目を統一フォーマットで調査し、`docs/01_data-source-survey.md` にまとめる。

### 4.1 調査対象と事前見立て

| データ | 情報源候補 | ライセンスの事前見立て（要検証） |
| --- | --- | --- |
| POI（飲食・文化・商業・ナイトライフ等） | Foursquare OS Places | Apache 2.0。商用利用可の見込み |
| POI・自然地物・歩行者エリア・旧市街 | OpenStreetMap | **ODbL。share-alike（派生DBの同条件公開義務）があり、他データとの組み合わせ制約が最大の論点** |
| 人口・標高・位置・施設属性 | Wikidata | CC0。制約ほぼなし |
| Wikipedia閲覧数（認知度） | Wikimedia Pageviews API | 無料・商用可の見込み。言語版（ja/en）ごとの取得方針を検討 |
| 世界遺産 | UNESCO World Heritage List | 再配布条件を要確認。Wikidata経由の代替も検討 |
| 月別気候（気温・降水・日照・湿度） | Copernicus ERA5（またはERA5派生の気候平年値） | 無料・商用可・帰属表示。取得の手間が大きいため Open-Meteo Climate API 等の派生サービスも比較 |
| 空港 | OurAirports | パブリックドメイン |
| 宿泊者数・観光客数 | Eurostat Tourism Database、各国統計 | CC BY 4.0。**ただし地域粒度（NUTS2等）と都市粒度のギャップ、小規模スポットのカバー率が論点** |
| 渡航安全情報 | 外務省 海外安全ホームページ | 政府標準利用規約。**国・地域単位のため154件への展開方法（国→都市への割り当て）が論点** |
| 都市間距離・空港からの距離 | 緯度経度からの計算（自前） | ライセンス問題なし。直線距離を基本とし、移動時間は将来課題 |

### 4.2 各ソースの調査項目（note.md指定の9項目）

1. 取得できる内容
2. 154件のカバー率（**実際にサンプル照会して実測する**。机上確認のみにしない）
3. 取得方法（API／一括ダウンロード）
4. 継続的・再現可能な取得可否
5. 更新頻度
6. 商用利用の可否
7. 帰属表示・再配布・加工の条件
8. 他データと組み合わせる際の制約（特にOSMのODbL）
9. 費用

### 4.3 成果物

- `docs/01_data-source-survey.md`：ソース別の調査結果＋「採用／代替採用／不採用」の判定と理由
- カバー率実測の根拠データ（`data/raw/survey/` に保存）

## 5. フェーズ2：データ構造設計（note.md 4.2）

調査結果と「最終的な提案体験からの逆算」に基づき、`docs/02_schema-design.md` とJSON Schemaを作成する。

### 5.1 設計方針

- **事実と指標を分離**する2層構造にする。
  - **Layer 1：事実データ**（実測値そのもの。例：美術館数=12、7月平均気温=24.3℃）
  - **Layer 2：派生指標**（提案ロジック用のスコア。例：文化施設集積度、季節別快適度、認知度）
  - 指標の算出式はドキュメント化し、事実データから再計算可能にする（「初心者向き」のような主観ラベルは付与しない、というnote.mdの方針に沿う）。
- 全レコードに**出所メタデータ**（source、取得日、ライセンス、帰属表示文）を持たせる。
- `type` により該当しない項目は null（例：フェリー航路にPOI密度は付けない）。

### 5.2 スキーマの想定ブロック（たたき台）

```
place
├── identity      … ID、名称（ja/en/現地語）、QID、国、type、緯度経度
├── geography     … 標高、人口、地形、近傍の自然地物（山・湖・海岸・展望地点）
├── poi           … カテゴリ別POI数・密度（飲食、美術館・劇場、商業、ナイトライフ、スパ、公園…）
├── heritage      … 世界遺産（件数・名称・分類）、歴史地区の有無
├── climate       … 月別の気温・降水量・日照時間・湿度（12か月×4指標）
├── access        … 最寄り空港（距離・種類）、主要都市からの距離、鉄道駅の有無
├── recognition   … Wikipedia閲覧数（ja/en）、宿泊者数・観光客数
├── safety        … 外務省危険情報レベル（国・地域単位からの展開）
├── indicators    … 派生指標（Layer 2）
└── meta          … 出所・取得日・ライセンス
```

### 5.3 質問回答 → 指標のマッピング表

note.md 3章（旅の醍醐味8種／旅のスタイル5種／旅の時期／旅行経験／自由記述）に対応する形で、「お客様の回答 → 使用する指標 → 使用する事実データ → データソース」の対応表を設計書に含める。ここが「提案体験からの逆算」の中核。

### 5.4 成果物

- `docs/02_schema-design.md`（設計書＋マッピング表＋指標算出式）
- `schema/place.schema.json`（JSON Schema）
- サンプル3件（大都市／小さな町／単体スポット、例：パリ・オビドス・シヨン城）を実データで作成し、設計レビューに使う

## 6. フェーズ3：154件のデータ作成（note.md 4.3）

### 6.1 やること

1. `scripts/` 配下に取得・変換スクリプトを作成する（**再現可能・継続更新可能**にするため、手作業コピペではなくスクリプト化する。note.mdの「継続的・再現可能に取得できるか」要件に対応）。
2. 実行パイプライン：取得（`data/raw/`）→ 正規化・結合（QIDキー）→ 検証 → 出力（`data/output/places.json`）
3. 派生指標（Layer 2）の算出。

### 6.2 品質検証

- **カバレッジ検証**：154件×主要項目の充足率マトリクスを作成（`docs/03_data-quality-report.md`）
- **妥当性検証**：既知の事実との突合サンプルチェック（例：パリの飲食店数が極端でないか、ツェルマットの標高、7月のセビリアの気温 等）
- **外れ値検出**：同typeのエントリ間で桁違いの値を機械的に抽出してレビュー

### 6.3 成果物

- `data/output/places.json`（154件・統一フォーマット）
- `scripts/`（取得・変換・検証スクリプト＋実行手順README）
- `docs/03_data-quality-report.md`（カバレッジ・検証結果・既知の制約）

## 9. リスクと対応

| リスク | 対応 |
| --- | --- |
| OSMのODbL（share-alike）が商用サービスでの他データ結合に影響する | フェーズ1で最優先に確認。問題があればPOIはFoursquare OS Places（Apache 2.0）＋Wikidata（CC0）中心に切り替える |
| 小さな町・単体スポットで観光統計（Eurostat等）が取れない | カバー率を実測し、取れない項目はnull＋代替指標（Wikipedia閲覧数等）で補う方針を設計書に明記 |
| 外務省安全情報が国単位でしか取れない | 国レベル値を各地点に展開し、粒度をメタデータに明記 |
| カタカナ名の同定ミス | フェーズ0で全件の同定一覧をレビューに掛ける |
| ERA5の生データ取得が重い | 平年値ベースの軽量な代替（Open-Meteo Climate API等）を比較検討 |

## 10. 未解決の確認事項（随時ご回答ください）

1. **航路2件の扱い**：`type: route` として最小属性のみ整備する想定でよいか（提案ロジック上「移動手段」としてどう使われる想定か）。
2. **曖昧な地名の同定**：フェーズ0完了時に一覧を提示するので確認をお願いしたい。
3. **更新運用の要件**：整備後のデータ更新頻度・担当の想定（スクリプトはいずれにせよ再実行可能に作る）。
4. **global-bm 側のインターフェース**：接客AIがこのデータをどう読み込む想定か（プロンプトに埋め込む／検索して渡す等）。設計には直ちに影響しないが、Layer 2指標の粒度設計の参考にしたい。
