# 都市提案ロジック用データ

# なぜやるのか？（Why）

現在の接客AI（β版）では、お客様の回答と154件の都市・観光地の一覧をLLMに渡して旅行先を提案しているが、提案判断の根拠となる客観的な事実データが十分に整備されていない。

JTB独自の知見は付加価値として重要である一方、それだけに依存すると、情報が多く蓄積されている都市に提案が偏り、情報の少ない都市・観光地を公平に比較できない可能性がある。また、客観的事実だけでは一般的な検索サービスとの差別化が難しい。

そのため、まずは施設・自然・気候・アクセス・認知度・安全情報など、定量的に確認できるデータを整備し、旅行先候補を網羅的かつ公平に比較できる基盤をつくる。そのうえでJTB独自の知見を組み合わせることで、提案の納得感とJTBならではの付加価値を両立させる。

# なにをやるのか？（What）

154件の都市・観光地を対象に、旅行先提案ロジックで利用できる客観的事実データを調査・設計・作成する。

具体的には、POI、自然資源、人口・標高・位置、Wikipedia閲覧数、世界遺産、気候、空港、観光統計、安全情報、都市間距離などのデータソースについて、取得可能な内容、カバー率、取得方法、更新頻度、商用利用条件、費用、他データとの組み合わせ制約を確認する。

調査結果をもとに、旅の醍醐味・旅のスタイル・旅の時期・旅行経験・自由記述の回答と結び付けられるデータ構造を設計し、154件分のデータを統一フォーマットで整備する。最終的には、飲食店密度、文化施設数、自然資源、気候適合性、認知度、アクセス性などを指標化し、提案ロジックに組み込める状態にする。

## 1. 目的

現在は、お客様の回答と154件の都市・観光地の一覧をLLMへ渡し、旅行先を提案しています。

お客様の希望に合う旅行先を、客観的な事実とJTBが持つ独自の知見を組み合わせて提案できる状態を目指します。

旅行先に関する情報には、大きく分けて次の二つがあると考えています。

- 施設、自然、気候、アクセス、認知度など、定量的に誰でも確認できる客観的な事実
- 現地の魅力、おすすめの過ごし方、知られざるスポットなど、JTBらしさや競争優位性につながる独自の知見

JTB独自の知見だけで旅行先を選ぼうとすると、情報が多く蓄積されている都市に提案が偏り、情報の少ない都市を十分に比較できない可能性があります。一方、客観的な事実だけでは、一般的な検索や他社サービスとの差別化が難しくなります。

そのため、まず客観的な事実をもとに、お客様の希望に合う旅行先を幅広く検討できる基盤を整えます。そのうえで、JTB独自の知見を組み合わせることで、提案の納得感と付加価値を高めます。

例えば、客観的な事実からパリがお客様の希望に合う候補として選ばれたうえで、「実は現地にJTBがおすすめするおいしいラーメン店がある」といった独自情報を加えることで、JTBならではの提案につなげるイメージです。

最終的には、客観的な事実による網羅性と公平性、JTB独自の知見による付加価値の両方を備えた旅行先提案の実現を目指します。

とここまでつらつら書きましたが、私の課題感は定量的に誰でも確認できる客観的な事実のデータ整備してないじゃん！ってことです。JTBの独自データは集めようとしていますが・・・

## 2. 想定しているデータ

客観的な事実のデータを整備するため、以下のようなデータを想定しています。

| データ | 情報源の候補 |
| --- | --- |
| 飲食店、文化施設、商業施設などのPOI | Foursquare OS Places、OpenStreetMap |
| 山、湖、海岸、展望地点、自然保護地域 | OpenStreetMap |
| 人口、標高、位置、施設属性 | Wikidata |
| Wikipedia記事の閲覧数 | Wikimedia Analytics API |
| 世界遺産の名称、位置、分類 | UNESCO World Heritage List |
| 月別の気温、降水量、日照時間、湿度 | Copernicus ERA5 |
| 空港の位置、種類、基本情報 | OurAirports |
| 宿泊者数、観光客数 | Eurostat Tourism Database、各国の観光統計 |
| 渡航上の安全情報 | 外務省 海外安全ホームページ |
| 都市間距離、空港からの距離 | 各地点の緯度経度、経路情報 |

上記は現時点での候補です。

データの品質、154件のカバー率、取得方法、更新可能性、**商用利用条件**を確認したうえで、より適切な情報源があれば、それでもOKです！

各データについては、少なくとも以下を確認してください。

- 取得できる内容
- 154件の対象をどの程度カバーできるか
- API、ダウンロードなどの取得方法
- 継続的・再現可能に取得できるか
- 更新頻度
- 商用利用の可否
- 帰属表示、再配布、加工の条件
- 他データと組み合わせる際の制約
- 費用（まずは無料から始めましょう）

## 3. データ活用の具体例

以下は、現在お客様から取得している回答と、どのような都市データを結び付けられそうかを示した例です。

### 3.1 旅の醍醐味

| お客様の回答 | 必要になりそうなデータ | 提案指標として検討できること |
| --- | --- | --- |
| 絶品グルメを味わう | 飲食店数、密度、料理カテゴリ、市場・食品店 | 都市規模に対する飲食店の多さや、食の多様性を比較する |
| 感性を揺さぶる体験 | 美術館、ギャラリー、劇場、コンサートホール | 文化施設の集積度と種類を比較する |
| 絶景に息をのむ | 山、湖、海岸、展望地点、標高差 | 自然地物の種類、都市からの距離、アクセス性を比較する |
| 最高の景色を写真に | 展望地点、歴史的街並み、水辺、自然地物 | 撮影対象の多様性と近さを比較する |
| 思いっきり体を動かす | ハイキングコース、スキー場、マリーナ、公園 | アクティビティの種類と季節適合性を比較する |
| 非日常にどっぷり浸る | 気候、地形、歴史地区、文化的背景、認知度 | 日本やお客様の日常環境との差を表す方法を検討する |
| 歴史を肌で感じる | 世界遺産、歴史的建造物、博物館、創建年代 | 遺産・史跡の密度や街の古さを比較する |
| 街を自由に散策 | 旧市街、歩行者エリア、カフェ、市場、商店 | 徒歩圏内に見どころがどの程度集積しているかを比較する |

### 3.2 旅のスタイル

| お客様の回答 | 必要になりそうなデータ | 提案指標として検討できること |
| --- | --- | --- |
| ワイワイ | ナイトライフ施設、飲食店、イベント施設 | にぎわいに関する特徴を比較する |
| ロマンチック | 展望地点、旧市街、湖畔、海辺、庭園 | 雰囲気に関係する都市環境を比較する |
| リラックス | スパ、温泉、公園、自然、施設密度 | 自然の近さや静かに過ごせる環境を比較する |
| ディスカバリー | Wikipedia閲覧数、宿泊者数、観光資源 | 認知度は低いが魅力のある都市を探す |
| ディープ | 特定分野の施設、地域固有の文化・産業 | 特定分野への特化度を比較する |

### 3.3 旅の時期

| お客様の回答 | 必要になりそうなデータ | 提案指標として検討できること |
| --- | --- | --- |
| 春・夏・秋・冬 | 月別の気温、降水量、日照時間、湿度 | 選択した季節に快適に過ごせる都市を比較する |
| おまかせ | 都市ごとの月別気候 | その都市に適した旅行時期を提案する |

現在は、LLMに対して「旬を考慮する」と指示しているのみで、判断の裏付けとなるデータがありません。

気候データを整備することで、旅行時期に関する提案を客観的な事実に接地させたいと考えています。

### 3.4 旅行経験

| お客様の回答 | 必要になりそうなデータ | 提案指標として検討できること |
| --- | --- | --- |
| ビギナー寄り | 認知度、宿泊施設数、空港からの距離、公共交通、安全情報 | 観光しやすい環境が整っている都市を探す |
| ベテラン寄り | 認知度、アクセス難易度 | 認知度の低い都市や、移動難易度が高い都市も候補に含める |

「初心者向き」といった主観的なラベルを直接付与するのではなく、アクセス性や観光基盤など、確認可能な事実を整備してください。

### 3.5 自由記述

| お客様の記述例 | 必要になりそうなデータ |
| --- | --- |
| パリには行きたい | 都市名称、国、地域などの基本情報 |
| 治安のよい街がいい | 外務省の安全情報など |
| 移動は少なめにしたい | 都市間距離、移動時間、空港からの距離 |
| 有名すぎない穴場に行きたい | Wikipedia閲覧数、宿泊者数、飲食店、宿泊施設、文化施設、自然 |
| 特になし、おまかせ | 追加データなし |

例えば「有名すぎない穴場」は、単に施設が少ない都市を意味するわけではありません。

将来的には、以下のような複数の事実を組み合わせて判断することを想定しています。

> 認知度は低い
> 
> 
> ＋ 希望との一致度が高い
> 
> ＋ 飲食・宿泊・文化・自然などが一定以上ある
> 

このような判断ができるよう、認知度と旅行先としての充実度を分けて確認できるデータを整備したいです。
